# 助手编排机制设计：照着「编码 Agent（我）怎么干活」来

- 日期：2026-07-03
- 状态：**待老板审阅**（审过再动手，先不写代码）
- 一句话：与其我凭空设计一个「理想机制」，不如把**我自己（编码 Agent）实际怎么干活**如实写下来，让 CRM 助手照着搬——因为我就是一个每天处理无限灵活指令、还能不出乱子的编排活样本。

---

## 缘起：为什么拿「我怎么干活」当蓝本

上一版文档更像「讲道理」，落到实现上偏虚，当指导不够硬。
最实在的参考不是抽象设计，而是一个**已经在正常工作的编排系统**——我就是。
你给我五花八门的要求（「修这个 bug」「合并这两段」「先查一下再改」），我**没有一张「要求分类表」**，照样能拆解、动手、纠错、完成。
所以这份文档：**先如实写我怎么编排，再一条条搬给 CRM 助手。**

---

## 第一部分：我（编码 Agent）实际是怎么编排的

如实讲，就 8 条：

1. **没有「先归类」这一步。** 我不会先把你的话归进某个固定类别再去查表。我直接读「你要什么 + 现场什么情况」，然后想第一步干嘛。
2. **核心就一个循环：想一步 → 干一步 → 看结果 → 再想。** 直到目标达成、或需要你拍板才停。**关键：计划不是一开始定死的，是边看边调的。**
3. **工具是积木，我自己拼。** 给我的是一批通用小工具（搜、读、改、跑命令…），没有「修 bug 工具」「重构工具」。任何复杂任务都是这几个小工具的**组合**——这就是我能应付没见过的要求的原因。
4. **动手前先摸清情况。** 改代码前我先搜、先读，弄懂了再动，不瞎改。（就是你说的「合并前先把两条记录拉出来对比」。）
5. **复杂活先列个待办清单，边做边勾。** 多步骤任务我维护一个清单当计划，做完一条勾一条；中途学到新情况就改清单。
6. **每一步都看结果，错了自己纠。** 改完就跑测试、查报错；挂了就找根因再改，而不是同一招硬试。（就是「核对结果对不对」。）
7. **危险的先问，可逆的直接做。** 删文件、改共享的东西这种不可逆动作，我先跟你确认；本地能撤销的我就直接干，不啰嗦。
8. **拿不准就问一句。** 信息不够，我问你要，而不是瞎猜、也不是硬报错。

> 压成一句：**开放地想怎么做（循环 + 组合积木），收紧地下手（危险动作先确认）。**

---

## 第二部分：现在 CRM 助手为什么不行

它恰恰把上面第 1、2 条给砍了。
它每句话都被塞进一张固定表格：**6 种对象 × 9 种任务** → 一个预设工具。只要要求不在表里，就选「最像的格子」，把不合的部分丢掉：

- 「今天要拜访的**客户**」→ 没有「按今天筛客户」的能力 → 只抓「客户」→ 列全部 9 家。
- 「**合并**这两条」→ 没有「合并」能力 → 塞进「改字段」→ 弹「要改哪个字段」。

人能说的动词无限（合并 / 去重 / 对比 / 拆分 / 顺延 …），这张表**永远补不完**。它没有「循环 + 组合」，所以每碰到新说法都得我手写规则——**这就是救火**。

---

## 第三部分：把我的做法搬给 CRM 助手（对照表）

| 我（编码 Agent）怎么做 | CRM 助手对应该怎么做 |
|---|---|
| 不先归类，直接想第一步 | 扔掉 6×9 分类表 |
| 想 → 干 → 看 → 再想 的循环 | 助手也走这个循环：列几步、一步步做、看结果再调 |
| 工具是积木，自己拼 | 用现有的 查/建/改/删 当积木，让模型自己拼（合并 = 查 + 对比 + 改 + 删） |
| 动手前先摸清情况 | 改之前先把相关记录查出来 |
| 复杂活先列待办清单 | 助手先列一个 2–4 步的小计划 |
| 每步看结果、错了自纠 | 出结果前核对「这数据真答上用户的问题了吗」 |
| 危险动作先问 | **改数据前先出确认卡** |
| 拿不准就问一句 | 缺关键信息就问用户，别瞎猜 |

**唯一比我收得更紧的地方**：我改的是本地代码（能撤销），CRM 助手改的是公司数据（不好撤销、你还被误改过一次）。所以对它，**「危险动作先问」是硬规矩：一切改数据都走确认卡。**

## 第四部分：所以 CRM 助手长这样（三个词讲完）

把我的循环落成最简单的形状：

1. **计划**：一个有序步骤清单（通常 2–4 步）。
2. **步骤只有两种**：
   - **查一下**：调一个现有工具（查 / 建 / 改 / 删）。
   - **想一下**：让模型对前面查到的东西做判断（对比、挑出、提方案）；后面步骤能看到前面结果。
3. **护栏**：凡是改数据的步骤，**先变确认卡给你看**，点了才动库。

（这就是我第 2、3 条循环的「够用版」：不做无限自动循环，需要看结果的地方用「想一下」步骤解决。）

### 走两个例子
- **合并两条重复拜访**：查这两条 → 想（提合并方案）→ 确认卡 → 改保留那条 + 删另一条。我们**从没写过合并工具**，它是积木拼的（正是你截图里 Copilot Studio 的流程）。
- **今天要拜访的客户**：查今天的拜访活动 → 想（列出涉及的客户）。不会再「列全部客户」。

**为什么治本**：覆盖面 = 步骤的**组合**，不是我预设了多少场景。去重、顺延、拆分、任何新说法都是同几块积木换个拼法，**零新增代码**。模型负责「怎么拼」，拼得好不好跟着模型能力涨。

## 第五部分：怎么落地（三小步，能分开验证、能回滚）

1. 第一个模型改成输出「步骤清单」，先只用现有工具串起来 —— 光这步，「合并」就能走通。
2. 加「想一下」步骤（后面步骤能看前面结果）—— 治「今天的客户」这类派生问题。
3. 所有改数据统一走「方案 → 确认 → 执行」，退役场景门。

全程用仓库里 `copilot-studio/eval-csv` 当回归集，保证老场景不退化。

- **复用**（现成的）：工具（查/建/改/删）、一步步执行的队列、确认卡（FormCard）。
- **新增**（很少）：让第一个模型吐「步骤清单」（换 prompt）+ 让执行器认识「想一下」（复用现在生成总结的那次模型调用，把前面步骤数据喂给它）。
- **退役**（做减法）：6×9 死映射、上轮我加的「要改哪个字段」场景门（就是它把合并搞坏的）。

---

## 第六部分：明确不做什么（保持简单）

- **不做**全自动、无限循环的 agent —— 只做「有序的几步」，看结果用「想一下」步骤。
- **不做**复杂的计划语言 / DSL —— 计划就是个简单的步骤数组。
- **先不做**并行、自动回滚重试 —— 顺序执行，简单可预测。

> 一句话原则：**开放地「列计划」，收紧地「改数据」。**

---

## 第七部分：确认卡的 generative UI（数据预览）

改数据前只列「更新 / 删除」这些动作,用户没法核对改得对不对。于是 `proposeChanges` 除了 `writes`(要执行什么),还返回一份**结构化数据预览** `followup`,前端**动态渲染**,让用户看清「改成什么样」。

**核心：大模型挑数据,前端挑渲染。**
- 大模型按意图返回数据段(**可选**,没有需要交互的结构化数据就不返回)：
  - `comparison`(前后对比) / `single`(单条记录,可标红) / `list`(多行表)。
- 前端一个通用渲染器按 `kind` 选组件——**固定、受审核的词汇表**,只渲染纯文本,认不得的段直接丢弃。
- **预览 ≠ 执行**：卡里显示什么,和真正落库的 `writes` 是**两份数据**;预览画歪也伤不到真实数据。
- **保真铁律**：对比里的 `after` 必须等于对应 write 的参数值(**所见=所写**),`before` 必须取真实原记录,**不许大模型自由誊写**(否则用户核对了个寂寞)。

覆盖面 = 数据段的**组合**：合并 = `comparison`(保留记录 前→后) + `single` danger(待删记录);去重 / 拆分 / 批量换个组合,**零新增代码**。只有真出现全新视觉(如画图)才加**一个**新段类型 + 组件,词汇表长得很慢、每个都受控。

> 这跟我(编码 Agent)给你看东西一样：我给的是结构化 diff / 代码块(固定展示词汇),不是我现画的 HTML。生成的是**内容与结构**,渲染用**固定安全的词汇表**。

## 附录：计划长什么样（给实现参考，老板可跳过）

「合并两条拜访」这份计划，内部大概就是这么个东西——一个数组，几个字段：

```jsonc
{
  "steps": [
    { "kind": "data",  "tool": "queryActivities", "args": { "dateRange": "today", "type": "visit" } },
    { "kind": "think", "task": "对比这两条，提出保留/合并/删除方案", "uses": [0] },
    { "kind": "data",  "tool": "updateActivity",  "fromThink": 1, "confirm": true },
    { "kind": "data",  "tool": "deleteActivity",  "fromThink": 1, "confirm": true }
  ]
}
```

- `kind`：`data`＝调工具，`think`＝让模型想。
- `uses` / `fromThink`：后面步骤引用前面步骤的结果。
- `confirm: true`：执行前必须弹确认卡。

比现在 `FrameResult`（对象 / 任务 / 时态 / relatesTo / explicitNames …）**更短、更好懂**。
