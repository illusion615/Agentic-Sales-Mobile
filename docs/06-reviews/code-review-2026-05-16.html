01 / Findings

先把已经闭环的和仍开放的分开看

本轮 review 的关键不是再找一堆毛病，而是确认云端到底修到了哪一步，剩下的坑还值不值得继续追。

一句话判断
当前最危险的问题已经不在“主链路会不会断”，也不在“会不会再长回第二套 hook”，而在“provider 既然成了唯一 owner，内部还能不能继续瘦下来，不把后续需求都拖慢”。
建议的修复顺序
1
继续把 copilot-context.tsx 往下拆，优先处理 transport / session 和 message persistence，visit extraction 已经先抽成 helper。
2
补一条围绕“新建会话、发送消息、visit extraction”的最小 smoke 回归链，让主链路收口后不再靠人工记忆守。
3
下一个热点不是首页小 lint，而是 function-executor.ts，它已经比 provider 更像后续功能堆积点。
Closed
copilot-service.ts 已补 freshness 校验，旧 token 不再盲复用
这条问题已经从“开放缺陷”变成“已在云端看到实现并同步回镜像”的状态。
+
问题是什么

旧实现只要从缓存里读到 conversation 就直接复用。只要 token 过期，用户第一次发消息就可能踩到失效会话。

证据
apps/sales-copilot/src/services/copilot-service.ts:15
apps/sales-copilot/src/services/copilot-service.ts:80-102
apps/sales-copilot/src/services/copilot-service.ts:237-256
现在变成了什么

会话对象现在带有 tokenAcquiredAt 和 conversationStartedAt。复用前会走 isConversationValid()，旧会话会被清掉再重建。

结论

这条已经可以从“待修”列表里拿掉。后续如果再出断链，优先看 provider 或页面调用路径，不要再回头怀疑 service freshness。

Closed
copilot-context.tsx 已停止在 streaming tick 上频繁落盘
云端当前实现已经把“UI 每帧更新”和“持久化落盘”拆开了。
+
问题是什么

原先每次 messages 变化都写一次 sessionStorage。而 streaming 模拟本身又会高频刷新消息内容，等于把同步存储绑到了每个 tick 上。

证据
apps/sales-copilot/src/contexts/copilot-context.tsx:276-345
apps/sales-copilot/src/contexts/copilot-context.tsx:276
apps/sales-copilot/src/contexts/copilot-context.tsx:297-301
现在变成了什么

当前实现使用 lastPersistedRef 去重，并且在有 isStreaming / isThinking 状态时直接跳过落盘，只在稳定 500ms 后再持久化。

结论

这条性能缺陷已经实质关闭。后面如果聊天面板还有卡顿，就该查渲染树和大组件重绘，而不是继续怀疑 sessionStorage 每帧写入。

Closed
首页和 visit-log 的页面级 Direct Line 编排已回收进 provider
这轮最大的结构收益，不是多了一个 helper，而是会话所有权终于收到了单一路径上。
+
问题是什么

之前首页自己建 conversation、自己轮询；visit-log 也自己发提取 prompt 和手写轮询循环。结果是同一种服务被三处实现各管一段。

证据
apps/sales-copilot/src/pages/home.tsx:1749-1763
apps/sales-copilot/src/pages/visit-log.tsx:394-398
apps/sales-copilot/src/contexts/copilot-context.tsx:198
apps/sales-copilot/src/contexts/copilot-context.tsx:382-390
现在变成了什么

首页新建会话直接走 copilot.startNewConversation()；visit-log 提取则走 copilot.extractVisitData()。页面不再自己抓 Direct Line 细节。

结论

主链路所有权已经回到 provider。剩下要防的是“又出现第四处私接 Direct Line”，不是继续拆旧页面轮询。

Open
copilot-context.tsx 仍然是新的超级节点，但已经拆出第一块 helper
这不是说 provider 不该做 owner，而是它虽然已经开始瘦身，体积和职责密度仍然偏高。
+
问题是什么

收口之后，provider 仍同时管面板开合、消息持久化、Direct Line 连接、流式输出、表单卡片状态和匹配选择。visit extraction 已经抽出，但主文件依然接近两千行。

影响

现在的风险不再是“会话是不是分叉”，而是“任何一处修改都必须读完整个 provider 才敢动”。即便已经拆出第一块 helper，这个阅读成本仍然会拖慢后续每个 Copilot 需求。

建议

保持 provider API 不变，下一步继续拆两块：transport/session、message persistence/streaming。现在已经证明“先抽 helper、外部不动”这条路能走通，就别再把剩余职责继续堆回 provider。

问题是什么

主链路已经回收到 provider 后，仓库里还留着一份旧的 use-copilot.ts。它即使暂时没人 import，也还是一个会把第二套状态机重新带回来的入口。

现在变成了什么

这份遗留 hook 已经直接删除，仓库里不再保留那套重复的 Direct Line 会话、轮询和发送逻辑。

证据

结论

这条已经可以从开放问题里拿掉。后面真正需要防的不是旧 hook 复活，而是别在别的页面或 helper 里重新长出新的私有 owner。

Closed
首页图标按钮已补可访问名称，剩余 inline-style debt 也已收掉

04 / Action Plan

建议的第一轮治理动作

这些动作都很具体，目标不是“更优雅”，而是防止主链路再次分叉。

动作 1：继续拆 provider 内部，不动 provider 外部 API

visit extraction 已经拆出去，下一步优先处理 transport / session 和 message persistence，两块都继续走“内部抽 helper、外部不改接口”的路子。

动作 2：补最小 smoke 回归链

覆盖“新建会话、发送消息、visit extraction”三步就够。目标不是全量测试，而是给刚收口并开始拆分的主链路立一条护栏。

动作 3：处理 function-executor.ts 热点

这个文件已经 1,449 行，比当前 provider 更像下一轮功能继续堆积的地方。优先把查询、写入、匹配和参数规范化拆层。

动作 4：补一份 Copilot owner 规则

把“哪些文件可以碰 Direct Line、哪些页面只能消费 provider 动作”写成短规则，避免后面又有人从别处私接会话。
