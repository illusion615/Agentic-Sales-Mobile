01 / 核心判断

脆弱性的根因：没有"契约"，只有"约定"

老板的直觉是准确的——上一次 base64 修复本身没有问题，不稳定来自架构层面。系统在三条关键缝合线上缺乏强制契约：错误如何跨层传递、页面级 AI 功能为何未纳入已有 skill 体系、LLM 的非结构化输出如何被驯服。三者都依赖开发者"记得这么写"，而非由类型系统或测试强制。任何一次修改只要踩中其中一条隐性约定，就会在远处引入新错误。

错误契约缺失

错误是字符串，不是类型

错误以 error?: string 在层间拼接传递（Frame failed: ${...}），丢失栈、丢失分类、丢失上下文。调用方无法按错误"种类"分支处理，只能正则匹配文案。

shadow-agent.ts#L226 · frame.ts#L391
LLM 调用未纳入 Skill 体系

8 处各自手搓 prompt 与解析

LLM 入口 invokeFlowForLLM 集中了传输与可用性检查（这是对的），但 6 个文件共 8 次调用各自拼 prompt、各自解析返回——系统已有 19 个 skill 的注册/执行体系，这 8 处却完全绕过。改一次输出格式，要同步改 8 处。

pages/*.tsx(5) · use-ai-summary-trigger，共 6 文件 8 次调用
输出契约脆弱

LLM 输出靠"修补"驯服

JSON 依赖 jsonrepair + 贪婪正则兜底；修补失败即回退原始串，错误延后到下游 JSON.parse 才爆发，定位困难。

power-automate-service.ts#L134 · agent-utils.ts#L280
一句话定性 这不是"某个 bug 没修干净"，而是系统把本应由类型与测试强制的契约，降级成了散落在各文件里的人工约定。健壮性赤字源于此。

02 / 分层与依赖

分层按功能域，不是单一直线栈

代码物理上分了 components / contexts / hooks / lib / services / generated 六个目录，但运行时实际是三个功能域并存，深度各不相同：常规功能域（列表/渲染）直达数据访问，页面级 AI 功能直连 Flow·LLM，只有对话式 Copilot 域才需要编排层。问题不在“越层”，而在：三个域靠目录约定而非物理隔离，页面级 AI 又未纳入已有 skill 体系。

命名考古：「shadow agent」到底是什么？——它已不是影子，而是唯一的生产路径

历史上 Frame 架构以「影子模式」与旧的 legacy 自由文本分类器并行运行，只记录、不决策，用于对比验证（shadow vs legacy benchmark），frame-shadow-viewer.tsx 是当时的人眼对比工具。影子模式转正后，文件已更名为 frame.ts / frame-viewer.tsx。

legacy 现已移除。shadow-agent.ts 的 runShadowPipeline 现在就是 Pass-1 意图识别的唯一生产路径——文件头注释自述 Frame (intents[]) → Orchestrator (DAG) → Executor、No second free-form classification call；copilot-agent.ts#L658 注释 Frame + Orchestrator drives production。

问题在残留：文件名/函数名仍叫「shadow」；compareFrameVsLegacy(frame, null, 0) 与 compareShadowVsLegacy(result, null, undefined) 永远与 null 比较，是恒空的死逻辑；recordBenchmark 的 legacy 侧全为占位；FrameShadowViewer 仍挂在面板上却只剩半边。命名与现实脱节会误导维护者以为系统跑着两套意图识别——详见下文 W9。

调用散点化

8 处 LLM 调用各管各的

activity-capture.tsx、home.tsx（含 3 次）、opportunity-review.tsx、visit-log.tsx、accounts.tsx、use-ai-summary-trigger 各自调用 invokeFlowForLLM 或其包装层 generateVoiceSummary。这些是独立功能域的直连，本身合理；但 8 处各自拼 prompt、各自解析返回，系统已有 19 个 skill 的注册/执行体系（suggestPlan 已证明 LLM-backed skill 可行），页面级 AI 功能却完全绕过它各自手搓——重试、限流、错误分类、格式修补都无法集中实施。

03 / 请求流水线

对话域的一条消息要穿过五层，每层各自吞吐错误

注：此链路仅适用于对话式 Copilot 域；常规功能域与页面级 AI 域不经编排层（见第 02 章）。用户一句话从 UI 进入，依次经过上下文 → 编排器 → Frame 意图识别 → 函数执行 → LLM/Dataverse。错误信息在每一层被捕获并重新拼接成新字符串，最终用户看到的是层层包裹后的文案，而非可定位的结构化错误。

04 / 健壮性缺陷清单

九项让系统"怕改动"的结构性弱点

以下按风险排序，每项附代码证据与整改方向。前三项是导致"改一处坏一片"的主因。

W1 错误处理无统一模型 已修复

原始证据：错误以字符串拼接跨层传递（"Frame failed: ${...}"），无栈、无分类、无上下文。

修复摘要：新增 lib/errors.ts，定义 AgentError 类型（type: 'llm'|'parse'|'dataverse'|'validation'|'transport'|'unknown' × layer: 'frame'|'orchestrator'|'executor'|'transport'|'context'）+ agentError() 构造函数 + rootCause() 链遍历 + toUserMessage() 用户友好映射（中英双语，不透传内部信息）+ toDevLog() 开发日志。PipelineResult.error 从 string 改为 AgentError；orchestrator.ts 3 处字符串拼接替换为 agentError()；copilot-agent.ts 出口处用 toUserMessage() 统一映射，用户不再看到 Frame failed: {giant JSON}，而是 智能服务暂时不可用，请稍后重试；开发者通过 console.warn(toDevLog(err)) 看到完整链。补 5 条单测验证构造/链/映射/判别。Build ✓ 2.01s，push 成功。

W2 LLM 调用散点化：8 处各自手搓 prompt/解析 已修复

证据：invokeFlowForLLM 被 5 个 page + use-ai-summary-trigger 共 6 个文件直接或间接调用（home.tsx 内含 3 次，其中 2 次经 generateVoiceSummary 包装层，合计 8 次），各自构造 prompt 并独立解析返回。注：copilot-context 不直接调用它，而是经编排层 copilot-agent。

定性：这些 page/hook 属于独立功能域（页面级 AI），直连 Flow·LLM 本身合理——不是越层穿透；真正的问题是系统已有完整的 skill 注册/执行体系（19 个 skill，suggestPlan 已证明 LLM-backed skill 可行），页面级 AI 却未纳入，各自手搓 prompt 和解析。

影响：一次输出格式或解析兜底的修改，需在 7 处重复实施，极易漏改；遥测与限流无法集中。

整改方向：将页面级 AI 功能注册为 skill，纳入已有 function-registry → skills-selector → function-executor 体系。每个 skill 声明 prompt 模板 + 输入参数 + 输出 Zod schema，executor 层统一实施错误分类、重试、限流、遥测。所有 skill 既可被对话域多意图推理调用，也可被页面直接调用——skill 是唯一的功能封装。
