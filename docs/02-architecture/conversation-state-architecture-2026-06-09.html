会话状态架构：实施规格

本文定义在现有 Frame · Plan · Execute 管线之上增加一个会话状态层（Conversation State）的完整实施方案。目标是让系统在多轮对话中稳定地：解析指代（"它 / 那个单子"）、延续未完成的任务、按确定规则判断何时复用已查数据、何时必须重新查询。

本文是实施依据。读完后，工程师应当无需再做架构推导即可开工；每一个会引起分歧的决定（状态归属、参数归一化、摘要生成、多意图提交顺序、阈值常量、范围边界）都已在文内给出明确结论，而非留给实现者自行判断。

文档关系： 本文延续 Agent Architecture v1.0（单轮推理链路），不替代它。v1.0 定义"一句话如何变成一次函数调用"；本文定义"多轮之间如何维护上下文"。v1.1/v1.2 为本文前身，已并入。

1 / 问题定义

当前架构是无状态的单轮命令翻译器

现有管线把每一句用户输入独立处理：识别意图、生成执行计划、调用函数、返回结果。它处理边界清晰的单轮请求很可靠，但处理跨轮延续的请求会失败，因为它不保留"对话当前所处的业务上下文"。

短窗口

历史只保留最近几条

Frame 取最近 4 条对话（frame.ts slice(-4)），Orchestrator 取最近 2 条（orchestrator.ts slice(-2)）。更早的信息被丢弃。

非确定复用

是否复用由 LLM 判断

Frame 输出 contextSufficient 决定是否复用上一条结果，且只看最近一条 recordList，不校验查询条件是否一致。

无目标概念

每轮都从零理解

系统目标是"完成一次函数调用"，不是"维护一个跨轮演化的任务"。补充信息会被当成新命令。

1.1 已发生的故障（用于回归验证）

现象直接原因根本原因
问"今天的任务"返回 0 条，但首页 Agenda 显示 5 条命中 contextSufficient 复用路径，recordList 是对象，Array.isArray 判 false 返回空数组短窗口 + 非确定复用 + 无结构化状态，三者叠加

结论：这不是某个日期过滤的孤立 bug，是架构缺少会话状态层导致的系统性问题。下文的方案以"这两个故障必须可复现地消失"为最低验收线。

2 / 目标与非目标

明确范围，避免实现时反复纠结

目标（In scope）

本次要做

  • 引入显式 ConversationState 对象，跨轮维护。
  • 指代消解：把"它 / 这个客户 / 第一个"解析到确定实体。
  • 确定性数据来源决策：复用 / 用页面数据 / 重新查询，由规则而非 LLM 决定。
  • 未完成任务的跨轮槽位补全（Pending Goal）。
  • 用结构化状态替代裸对话文本喂入 Prompt。
非目标（Out of scope）

本次明确不做

  • 不做全自主 agent（开放工具、模型自由规划）。原因：当前 LLM 后端为无状态单次调用，且平台 JSON 输出模式不可靠。
  • 不改动写操作确认流程（draft / update 仍走确认卡片）。
  • 不改动 weekly-report、Copilot Studio 知识问答等旁路链路——它们不纳入会话状态层，保持现状。
  • 不引入服务端会话状态；P5/P6 状态仅存于前端。
3 / 状态对象定义

ConversationState 数据结构

ConversationState 是每轮消息处理前读取、处理后写回的结构化对象。它不是聊天记录全文，而是系统对当前对话业务上下文的显式描述。

type EntityType = 'account' | 'contact' | 'opportunity' | 'activity';

type ConversationState = {
  focus: FocusEntity[];        // 当前对话涉及的实体，按置信度排序
  workingSets: WorkingSet[];   // 最近若干次查询的结果集缓存
  pendingGoal?: PendingGoal;   // 未完成的创建/更新任务，至多一个
  rollingSummary: string;      // 超出窗口的历史的压缩摘要
  lastUpdatedAt: number;       // epoch ms
};

type FocusEntity = {
  type: EntityType;
  id?: string;                 // 已解析到 Dataverse 记录时存在
  name: string;
  confidence: number;          // 0–1，每轮衰减，见 §5.4
  source: 'page' | 'query-result' | 'user-mention' | 'created-record';
  turnIntroduced: number;      // 进入 focus 的轮次序号，用于衰减计算
};

type WorkingSet = {
  id: string;                  // uuid
  entity: EntityType;
  sourceFunction: string;      // 如 'queryActivities'
  argumentsHash: string;       // 见 §5.1，复用判定的唯一依据
  filterSummary: string;       // 人类可读的过滤条件，仅用于展示/Prompt
  records: Array<{ id: string; title: string; summary?: string }>;
  createdAt: number;
  stale: boolean;              // 超过 TTL 或被写操作失效后置 true
};

type PendingGoal = {
  fn: string;                  // 目标函数，如 'draftActivity'
  state: 'OPEN' | 'FILLING' | 'READY' | 'COMMITTED' | 'ABORTED';
  requiredSlots: string[];     // 必填字段名
  filledSlots: Record;
  nextQuestion?: string;       // 指向第一个缺失槽位的追问文案
  turnOpened: number;
};
// ⚠ 注意：上述 PendingGoal 为 v1.3 初稿语义，已在 §8 经实施驱动重设计否决。
// 最终语义见 §8.3——pendingGoal 收敛为指向未完成卡片的三态跨轮指针
// （OUTSTANDING / SUSPENDED / RESOLVED），不再是独立的逐轮问答状态机。
字段解决的问题判定要点
focus指代消解"它 / 这个客户"解析到 focus，而非让 LLM 从短历史猜测。
workingSets结果集延续与复用复用必须比较 argumentsHash，不能只比较实体类型。
pendingGoal跨轮任务缺字段的创建/更新任务可等待补全，不被当成新命令。
rollingSummary长历史压缩超窗口的业务事实进摘要，token 与对话长度解耦。
4 / 状态归属与线程模型

状态放在哪、谁更新、如何跨轮传递

这是实现前必须先定的问题。结论如下，实现时不要偏离。

4.1 归属

  • ConversationState 由 CopilotContext（src/contexts/copilot-context.tsx）持有，使用 useRef<ConversationState>（命名 conversationStateRef），与现有 messagesRef 并列。
  • 用 ref 而非 useState：状态在一次 sendMessage 内被同步读写多次，ref 避免闭包读到旧值（与 messagesRef 现有做法一致）。需要驱动 UI 的部分（如 Frame Inspector）另用 useState 镜像。
  • 会话 id 当前在 home.tsx 的 currentConversationId。状态对象不依赖它；P6 持久化时由 copilot-conversation-log.ts 负责把状态摘要与会话 id 关联，不把状态散落到 home.tsx。

4.2 调用点与签名

状态在一次消息处理中的流动路径固定为：hydrate（处理前）→ 传入 processMessage → resolve/execute → commit（处理后）。以下为实际已接线的代码（2026-06-11）。

// copilot-context.tsx · sendMessage 内（已接线）
const hydratedState = hydrateConversationState({
  prevState: conversationStateRef.current,
  pageContext: pageContextToFocus(pageContextRef.current), // PageContext → FocusPageContext 映射
  turn: messagesRef.current.length,
});
conversationStateRef.current = hydratedState;

const response = await processMessage(text.trim(), {
  userId, userEmail, locale, conversationHistory,
  lastFunctionResult, lastFunctionCalled,
  pageContext: /* 现状不变 */,
  state: hydratedState,        // 新增：传入快照（只读）
});

// processMessage 不直接修改 state；它把"应如何变更状态"作为结果返回。
// 灰度期 mutation 可能为 undefined，故用 ?? {} 兑底（commit 为 no-op）。
conversationStateRef.current = commitConversationState(
  conversationStateRef.current, response.stateMutation ?? {});
// copilot-agent.ts · processMessage 入参与返回新增字段（均可选，便于分阶段灰度）
type ProcessContext = { /* ...现有字段... */ state?: ConversationState };
type AgentResponse = { /* ...现有字段... */ stateMutation?: StateMutation };

// StateMutation 描述本轮发生了什么，由 commit 负责落到 state，
// 使 processMessage 保持纯粹（不持有 React 引用，便于单测）。
type StateMutation = {
  executedFunction?: string;
  executedArgsHash?: string;
  filterSummary?: string;               // 人读过滤描述，仅展示/Prompt
  resultRecords?: Array<{ id: string; title: string; summary?: string }>;
  createdRecord?: { type: EntityType; id: string; name: string };
  resolvedFocus?: FocusEntity[];
  pendingGoal?: PendingGoal | 'CLEAR';
  invalidatedEntities?: EntityType[];   // 写操作后需失效的 working set
  summaryNote?: string;                 // 失败/事实备注，追加入 rollingSummary
};

4.3 并发

用户连发消息时，状态更新必须串行。sendMessage 已有发送中的判断；在此基础上保证 commitConversationState 按消息到达顺序执行，不允许两次 commit 交叉。

4.4 重置

startNewConversation（copilot-context.tsx ~L2036）必须在清空 messages 的同一处把 conversationStateRef.current 重置为 emptyState()，避免跨会话串味。

5 / 状态生命周期

hydrate · commit · 归一化 · 衰减

5.1 argumentsHash 与 normalizeArg（复用判定的核心）

working set 能否复用，唯一依据是"本轮查询的有效参数集是否与缓存一致"。argumentsHash 只纳入影响结果集的参数，排除仅影响展示的参数（limit、sortBy）。归一化规则必须固定，否则同义输入会产生不同 hash。

function computeArgumentsHash(fn: string, args: Record): string {
  const SIGNIFICANT: Record = {
    queryActivities:   ['type','accountId','dateRange','scheduledDate','dateFrom','dateTo','status'],
    queryOpportunities:['stage','accountId','minAmount','minConfidence','maxConfidence','closingWithinDays'],
    queryAccounts:     ['name','region','tier'],
    queryContacts:     ['accountId','name','title'],
  };
  const keys = SIGNIFICANT[fn] ?? [];
  const norm = keys
    .map((k) => [k, normalizeArg(k, args[k])] as const)
    .filter(([, v]) => v !== undefined && v !== '');
  return fn + '|' + JSON.stringify(Object.fromEntries(norm));
}

normalizeArg 规则（必须按此实现，不得自由发挥）：

参数类别归一化规则示例
枚举值（stage / type / status）转小写、去首尾空格；映射已知别名到标准值"Negotiation" → "negotiation"
相对日期（dateRange）解析为绝对区间 [startISO, endISO]，按用户本地时区的当日 00:00–23:59"today" → ["2026-06-11T00:00","2026-06-11T23:59"]
绝对日期（scheduledDate / dateFrom / dateTo）截断到日（YYYY-MM-DD），去掉时间与时区后缀"2026-06-11T08:00Z" → "2026-06-11"
id（accountId 等）转小写（Dataverse GUID 大小写不敏感）"A1B2…" → "a1b2…"
金额 / 数值转 number；单位换算（200k→200000、50万→500000）"200k" → 200000
名称模糊词（name / title）转小写、去首尾空格，不做分词" Royal London " → "royal london"
这是"今天任务"故障的修复点。 "今天的任务"带 dateRange=today，归一化为当日绝对区间后参与 hash；历史 working set 若来自"全部活动"（无该过滤），hash 不一致 → 判定必须重查。仅比较"实体类型相同"不足以防止该故障。

5.2 hydrate（处理前重建）

function hydrateConversationState(input: {
  prevState?: ConversationState;
  messages: ChatMessage[];
  pageContext?: PageContext;
  turn: number;
}): ConversationState {
  const state = input.prevState ?? emptyState();
  // 1. focus：合并三个来源，按置信度降序，去重（同 type+id 取高者）
  state.focus = mergeFocus([
    fromPageContext(input.pageContext, input.turn),       // source='page'
    ...latestSingleEntityAsFocus(state.workingSets),      // source='query-result'
    ...decayFocus(state.focus, input.turn),               // 旧 focus 衰减后保留
  ]);
  // 2. working set 新鲜度：超 TTL 置 stale（不删除，仍可用于指代）
  state.workingSets.forEach((ws) => {
    if (Date.now() - ws.createdAt > WORKING_SET_TTL_MS) ws.stale = true;
  });
  // 3. rollingSummary：把溢出窗口的轮次压缩进摘要（§5.5 启发式，不调 LLM）
  state.rollingSummary = buildRollingSummary(state.rollingSummary, overflowTurns(input.messages));
  state.lastUpdatedAt = Date.now();
  return state;
}

5.3 commit（处理后写回）

function commitConversationState(state: ConversationState, m: StateMutation): ConversationState {
  // 查询：同 hash 覆盖、异 hash 追加；每实体最多保留 MAX_WORKING_SETS_PER_ENTITY 组
  if (m.executedFunction && isQuery(m.executedFunction) && m.resultRecords) {
    upsertWorkingSet(state, {
      entity: entityOf(m.executedFunction),
      sourceFunction: m.executedFunction,
      argumentsHash: m.executedArgsHash!,
      records: m.resultRecords,
      createdAt: Date.now(), stale: false,
    });
  }
  // 写操作后：使相关实体的 working set 失效（与 invalidateRelatedQueries 对齐）
  (m.invalidatedEntities ?? []).forEach((e) =>
    state.workingSets.filter((w) => w.entity === e).forEach((w) => (w.stale = true)));
  // 创建：新建记录成为高置信度 focus
  if (m.createdRecord) pushFocus(state, { ...m.createdRecord, confidence: 0.95, source: 'created-record' });
  if (m.resolvedFocus) state.focus = mergeFocus([...m.resolvedFocus, ...state.focus]);
  if (m.pendingGoal === 'CLEAR') state.pendingGoal = undefined;
  else if (m.pendingGoal) state.pendingGoal = m.pendingGoal;
  state.lastUpdatedAt = Date.now();
  return state;
}
失败语义（必须实现）： 若 execute 失败，StateMutation 不携带 resultRecords，commit 不写 working set，也不动 focus；只允许写一条 rollingSummary 失败备注。即"失败不污染状态"。

5.4 focus 衰减

每经过一轮，confidence ×= FOCUS_DECAY。低于 FOCUS_MIN_CONFIDENCE 的 focus 不再用于自动指代（必须改为追问或重新解析），但保留在数组中直到被淘汰。新建记录、页面绑定实体每轮重新注入，因此不会被衰减淹没。

5.5 rollingSummary 生成（P5 用启发式，不调 LLM）

为与"确定性"和"后端不稳定"两个约束一致，P5 的摘要不调用 LLM，按固定模板拼装；只在 P7 之后再评估是否引入 LLM 摘要。

// 仅保留业务事实，丢弃寒暄。模板化拼接，长度上限 ROLLING_SUMMARY_MAX_CHARS。
function buildRollingSummary(prev: string, overflow: ChatMessage[]): string {
  const facts: string[] = [];
  for (const m of overflow) {
    if (m.recordList) facts.push(`查询${entityLabel(m.recordList.type)}：${m.recordList.records.length} 条`);
    if (m.createdRecord) facts.push(`新建${entityLabel(m.createdRecord.type)}「${m.createdRecord.name}」`);
    // 用户消息只在包含实体名/金额/日期时保留要点，否则丢弃
  }
  const merged = dedupe([prev, ...facts]).join('；');
  return merged.length > ROLLING_SUMMARY_MAX_CHARS
    ? merged.slice(merged.length - ROLLING_SUMMARY_MAX_CHARS)  // 保留最近的事实
    : merged;
}
6 / 数据来源决策

复用 / 页面数据 / 重新查询，由规则决定

是否复用不由 LLM 单独决定。Frame 仍可输出 contextSufficient 作为"建议"，但最终来源由运行时 resolveDataSource(step, state) 按确定规则给出。

type DataSource = { kind: 'reuse'; workingSetId: string }
                | { kind: 'page' }
                | { kind: 'requery' };

function resolveDataSource(step: PlannedStep, state: ConversationState): DataSource {
  if (!isQuery(step.fn)) return { kind: 'requery' };
  if (step.userRequestedRefresh) return { kind: 'requery' };   // "重新查""刷新"
  const hash = computeArgumentsHash(step.fn, step.args);
  const ws = state.workingSets.find(
    (w) => w.sourceFunction === step.fn && w.argumentsHash === hash && !w.stale);
  if (ws) return { kind: 'reuse', workingSetId: ws.id };
  if (pageCovers(step, state /* pageContext */)) return { kind: 'page' };
  return { kind: 'requery' };
}
场景决策依据
商机列表之后问"哪个金额最高"reuse同函数、同 hash、working set 未 stale；只做分析
历史有"全部活动"，本轮问"今天的任务"requery本轮带 dateRange=today，hash 不一致
用户说"刷新一下"requery显式要求刷新，跳过复用
"继续刚才那个客户"不走来源决策这是指代消解（§7），不是结果集复用
"把它标记完成"不复用，走写流程写操作；需 focus 置信度达标 + 确认卡片（§8）
7 / 指代消解

把指代词解析到确定实体

"它 / 这个客户 / 那个单子 / 他们 / 第一个"先用确定规则解析；规则无法判定时才交给 LLM，并把候选集明确喂给它。解析结果必须可追溯（记录解析到哪个实体、依据哪条规则），展示在 Frame Inspector。

表达解析顺序失败处理
"它 / 这个 / 那个" + 实体类型明确① 同类型 focus（confidence ≥ FOCUS_MIN_CONFIDENCE）② 最近 working set 的单一记录追问并列出候选
"它"但类型不明确① 仅有一个达标 focus → 用它 ② 多个 → 取置信度最高且唯一最高者 ③ 并列最高 → 追问追问"你指的是哪一个？"
"第一个 / 第 N 个 / 最贵的那个"对最近 working set 的 records 按对应维度取值working set 已 stale → 先重查再取
"他们 / 这些"整个最近 working set（集合级指代）无 working set → 当作新查询
"继续刚才那个客户"account 类型 focusfocus 衰减过低 → 追问
类型不明确是难点，已在上表第 2 行给出确定规则。 不允许实现成"直接交给 LLM 猜"——只有在多个 focus 置信度并列最高、规则无法裁决时，才追问用户，而不是猜测。
8 / 跨轮补全未完成的草稿

草稿挂起时，下一句补充能接回草稿，而不是被当成新命令

8.1 要解决的问题

当用户发起一个创建/更新任务但信息不全时，系统会弹出一张草稿卡等待补全。问题出在用户不点卡片、而是直接用一句话补充的时候——这句补充目前会被当成一条全新命令处理，草稿卡被晾在一边。

目标：让"明天下午"被识别为对那张草稿卡日期字段的补充，填进卡片，用户再确认保存。整个过程不引入逐轮问答，只补上"草稿挂起时接住下一句补充"这一个缺口。

8.2 为什么现在补不上

系统对"信息不全"已有两类卡片来承载待补内容，用户在卡片上补全后保存——并不依赖一问一答。

缺什么卡片用户怎么补
缺字段（如拜访缺日期）草稿卡 formCard在卡上编辑字段 → 保存 / 取消
实体认不准（如有多个"John"）澄清卡 awaitingClarification点选候选，或回"新建 / 换名 / 跳过"

关键差别在于：消息进来时，系统只对澄清卡设了一道"跟进门"——会把下一句路由去补这张卡；草稿卡没有这道门，于是补充被漏给了 LLM 当新命令。这就是要补的缺口。

8.3 设计：用一个跨轮指针记住"有一张草稿待补"

引入 pendingGoal——一个指向当前未完成卡片的轻量指针。它本身不存卡片里的字段值（真值始终在卡片上），只记录"现在有一张什么卡、在哪条消息上、还缺哪些字段"，用来支撑两件事：跨轮接住补充、跑题后能提示回到草稿。

type PendingGoal = {
  fn: string;                    // 目标函数，如 'draftActivity'
  cardMessageId: string;         // 指向那张草稿卡的消息 id（真值在卡上）
  entityType: EntityType;        // 用于跑题判定与跨轮指代
  missingSlots: string[];        // 还缺哪些字段（只读快照，仅用于判定/提示）
  status: 'OUTSTANDING' | 'SUSPENDED' | 'RESOLVED';
  turnOpened: number;
};

它只有三个状态，且状态的变化由卡片自身的事件驱动（渲染、用户跑题、保存/取消），指针自己不"做主"：

不复制卡片数据： 需要补全或展示时，用 cardMessageId 回查卡片本身。missingSlots 只是渲染那一刻的只读快照，用于判断"下一句像不像字段值"和提示文案，不作为保存依据——避免出现两份会各自漂移的数据。

8.4 跟进门：怎么判断下一句是"补充"还是"新命令"

这是本设计唯一改变运行时行为的地方。在已有的澄清卡跟进门之后、LLM 管线之前，加一道判定：若当前有一张待补草稿，且下一句"看起来是字段值"，就补进卡片；否则照常当新命令处理。

判定"像字段值"必须三条全部满足（从严，宁可漏判也不误吞）：

判据含义反例（判为新命令）
无动词意图不含"查/列/新建/更新/删除/帮我…"等功能谓词"帮我查一下他的订单"
可解析为缺失字段的类型缺日期时"明天下午/周五/6月15日"可被日期解析器识别；缺金额时为可解析数值"随便看看"（解析不出任何缺失字段）
未引入新实体名不含与当前草稿无关的新客户/联系人名"改成给 Mayo 登记"
跑题时不丢草稿： 当下一句被判为新命令，不补卡——把 pendingGoal 置 SUSPENDED，正常执行新命令，并在回复尾部附一句轻提示"你还有一张未完成的草稿，要继续吗？"。草稿只有被用户保存/取消才清除，不因跑题而消失。
安全护栏： 跟进门只对草稿（draft* 系列）生效；查询/更新不进此路径。任何解析存疑一律放行到 LLM 管线——宁可让用户重述一次，也不误吞正常消息。补卡只改卡片字段，最终保存仍需用户在卡上点确认，写操作护栏不变。

8.5 生命周期同步与边界

指针必须与卡片严格同步，否则会指向一张已消失的卡。规则固定如下：

时机动作
渲染草稿/澄清卡写入 pendingGoal = { status:'OUTSTANDING', cardMessageId, ... }
卡片 保存 / 取消 / 跳过置 RESOLVED 并清除（三处终态回调接同一清除函数，避免遗漏）
又来一张新草稿卡旧的先置 RESOLVED（视为放弃），同一时间至多一个 pendingGoal
开新会话随 emptyState() 一并清除（§4.4）
连发两条补充逐条判定、各补一个字段；补全后仍待用户保存，不自动提交
补充值非法（如日期是过去）写入字段并交由卡片现有校验提示，不静默吞

P5 不持久化 pendingGoal——卡片本身不跨会话存活，指针也就没有持久化意义。

8.6 多意图一句话拆多步时的状态提交

系统支持一句话拆成多步（DAG，intent-queue-runtime.ts）。会话状态的提交规则：

  • 按拓扑顺序逐步执行，每步成功后立即提交自己的状态变更，后续步骤才能读到前序产生的 focus / working set。
  • 某步失败：该步不提交（失败不污染状态，§5.3）；依赖它的后续步跳过，不依赖的继续。
  • 一句话内多个新建记录：最后一个成为最高置信度 focus，其余按创建顺序保留。
  • 一句话内多个查询：各自按 hash 独立写入 working set，互不覆盖。

一条端到端 trace（状态逐轮演变）

本节用一段真实对话展示状态如何变化，作为"设计者意图"的标准答案。实现产生的状态轨迹应与此一致。省略号表示与上轮相同。

轮1  用户：「Cleveland 的商机进展如何」
  resolveDataSource → requery（无匹配 working set）
  执行 queryOpportunities{accountName:"Cleveland"} → 4 条
  commit:
    workingSets += { fn:queryOpportunities, hash:"queryOpportunities|{...accountId:c...}", records:4, stale:false }
    focus = [ account "Cleveland Clinic"(0.9, query-result) ]

轮2  用户：「哪个金额最高」
  指代/分析：同实体追问；resolveDataSource → reuse（hash 与轮1一致、未 stale）
  不重查，从 working set 取 amount 最大 → 回答
  commit: focus 衰减 account→0.72；新增 opportunity focus(0.9)

轮3  用户：「今天有哪些任务」
  resolveDataSource → requery（queryActivities，dateRange=today，无匹配 hash）
  执行 queryActivities{dateRange:today} → 5 条   ← 故障场景：此处必须重查，不得复用
  commit: workingSets += { fn:queryActivities, hash:"...dateRange:[今日区间]", records:5 }

轮4  用户：「帮我给第一个客户登记一次拜访」
  指代「第一个客户」：此时 account focus 已衰减至 0.9×0.8³≈0.46，低于 FOCUS_MIN_CONFIDENCE，不再自动指代
  → agent 将「第一个客户」重新解析为会话主体 Cleveland Clinic（非取轮1 商机 working set——它是 opportunity、不携 account）
  意图 draftActivity，缺 scheduledDate → pendingGoal OPEN，追问日期
  commit: resolvedFocus += account Cleveland Clinic(0.9, user-mention)；pendingGoal = { fn:draftActivity, state:OPEN, requiredSlots:[date], filled:{account} }

轮5  用户：「顺便看看 Mayo 的联系人」（跑题）
  本轮意图 queryContacts 与 pendingGoal(draftActivity) 不一致且无可填槽位
  → 挂起 pendingGoal（不清除），先执行 queryContacts
  commit: workingSets += contacts；pendingGoal 保持 OPEN

轮6  用户：「明天下午」
  本轮可填 pendingGoal 的 scheduledDate → FILLING→READY
  生成确认卡片（写操作护栏），等待确认
  commit: pendingGoal.state = READY, filled:{account, date:明天}
