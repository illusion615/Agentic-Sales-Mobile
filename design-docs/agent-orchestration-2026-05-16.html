00 / Context & Scope

Sales Copilot Mobile 的 agent orchestration 是一条短链路、单轮决策的智能表单流水线，本文记录其当前架构并基于真实代码评审其结构性问题。

系统主要承担三类工作：将自然语言请求转为对 CRM 的结构化操作（查询 / 草稿 / 更新）、在记录创建前提供可编辑的 form card 供用户确认、并对模糊匹配场景提供候选选择。完整链路由 Copilot Panel → Copilot Context → Copilot Agent → Function Executor → Form Card → Data Layer 构成；Copilot Studio 作为旁路工具（产品知识、外部知识、回退）通过 Direct Line 接入，不在主决策路径上。

本文范围限定在该 app 内的 orchestration 与其与 Copilot Studio 的当前边界关系；不覆盖通用多智能体平台设计、UI 视觉规范、以及 Copilot Studio 后端蓝图。

02 / Agent Decision Model

单轮 LLM 决策的设计、能力边界与缺陷矩阵

Agent 层（copilot-agent.ts）对每条用户消息执行一次主决策：一次 LLM 调用，输出一份结构化 JSON。本节披露这次调用的输入构成、输出契约、规则集合，以及当前能力的全景矩阵。

决策机制：单次 LLM 调用
输入 = 用户消息 + 历史 + pageContext + 函数清单 + 16 条规则。

规则与示例以中文/英文 prose 拼入 system prompt（copilot-agent.ts:313–520），由 LLM 一次性产生 function、arguments、matchTarget、additionalActions、multiIntentAnalysis 等所有字段。无后续推理回合。

输出契约：IntentResult JSON
由 Zod schema 强制（agent-utils.ts:57）。

主字段：function、arguments、requiresMatching、matchTarget（单实体）、additionalActions[]、multiIntentAnalysis、directResponse。解析失败 → fallback Copilot Studio + 30s 熔断。

分流前置：两条快速通道
在 LLM 调用前做硬路由（copilot-agent.ts:301–325）。

① 产品页 + 关键词正则 → 直送 Copilot Studio；② 熔断打开或未配置 BYOM → 直接错误返回。除此以外一律进入单轮 LLM 决策。

05 / Architectural Issues

七项架构问题，按严重度排列

每项给出：症状（用户视角） → 触发机制（运行时） → 架构根因（结构层） → 影响 → 代码锚点。严重度按“是否阻断核心场景”而非“是否常见 bug”定义。
