Issue I-8 · 设计

从"做什么"到"什么时候做、是否做完、要不要顺手提单"

当前 Activity 草稿只关心实体和标题，忽略了用户措辞里的时态、模态、极性，也不主动把对话中的购买信号沉淀为 Opportunity。 I-8 引入 temporalMode 字段与 Opportunity 自动发现卡， 让 agent 能区分"计划"与"已完成"、并在保存活动后给用户三选一的商机沉淀入口。

01 / 问题与目标

当前 agent 听不见"已经/即将"，也接不住购买信号

I-8 收敛两个相关但独立的缺口：时态盲与商机沉淀盲。两者共享同一次用户表达，但落到不同的字段和不同的交互卡片。

缺口 1时态盲：planned 与 completed 都被当成 open +

无论用户说"明天去拜访"还是"刚拜访完"，Activity 一律以 status=open + 今天的日期落库。下游 dashboard 把它们混在一起，"已完成拜访数"统计失真，销售必须事后手动改状态。

用户表达当前落库用户期望
"明天要拜访 King's College"status=open, date=今天status=open, date=明天
"刚拜访完 Rachel"status=open, date=今天status=completed, date=今天
"拜访 King's College"（裸句）status=open, date=今天同当前（缺省合理）
缺口 2商机沉淀盲：对话里飘过的购买信号没人接 +

用户记录拜访时常带出"客户想要 demo / 预算 80 万 / Q3 上线"这类购买信号。当前 agent 把这些信息塞进 activity 备注，但不会主动建议建立 Opportunity，导致 pipeline 漏录。

预期：activity 保存成功后，若识别到购买信号 → 弹出 Opportunity 检测卡，用户三选一：跳过、挂到已有 opp、新建 opp。

目标Done 的判定 +
  • 时态识别准确率：明确句式（含 just / will / tomorrow / 了 / 要）≥ 95%，歧义句式（裸说"拜访 A"）走 unspecified，行为不变。
  • 商机检测召回：含明确金额或时间窗的对话 ≥ 80% 触发卡片；裸说"客户感兴趣"不触发（避免骚扰）。
  • 零回归：完整跑通 I-1、I-2、I-3 现有 E2E 场景结果不变。
02 / Slice A · 时态感知

用语义字段 temporalMode 把语言学复杂度收口在 prompt

不让 LLM 输出"过去时/将来时"这种语言学标签，只输出语义意图：planned / completed / unspecified。Context 端用确定性映射表落到 status 与字段可见性，可单测、可调阈值。

SchemaIntentResult 新增字段 + 
// agent-utils.ts
const TemporalModeSchema = z.enum(['planned', 'completed', 'unspecified']);

IntentResultSchema = z.object({
  // ...existing
  temporalMode: TemporalModeSchema.optional(),  // 缺省视为 unspecified
});

可选字段、缺省视 unspecified，老对话历史与现有 Vibe 后端无需迁移。

PromptRule 9 · 时态判定（zh + en，各 3 例） +

在 system prompt 的"意图判定规则"段加 Rule 9，与现有 Rule 8b（cascade）平级：

9. temporalMode 判定（仅对 Draft Activity 与 Update Activity 意图生效）：
   - planned: 用户明确表示动作发生在未来。线索: 明天/下周/will/going to/scheduled。
   - completed: 用户明确表示动作已发生。线索: 刚/已经/了/just/finished/did。
   - unspecified: 无明确时态线索，或同一句中存在矛盾。

   Examples (zh):
   - "明天要拜访 Rachel" → planned
   - "刚拜访完 Rachel" → completed
   - "拜访 Rachel" → unspecified

   Examples (en):
   - "Plan a meeting with Rachel tomorrow" → planned
   - "Just visited Rachel, went well" → completed
   - "Meeting with Rachel" → unspecified

    name, accountName, productName, estimatedValue, ...,

04 / 置信度算法（已改为 LLM 自评分）

权重表写进 prompt，LLM 一次性识别 + 评分 + 决定是否输出

原方案让 LLM 只指认信号、Context 端打分。实施时发现：① 多打一次 prompt 浪费 token；② 权重表硬编码很难覆盖中英文+隐喻+多语种表达。改为把权重表写进 copilot-agent.ts Rule 6，LLM 一次性产出 type+quote+总分，<40 直接不输出 draftOpportunity additionalAction。

信号类型权重识别线索举例
amount+30£800K · 80万 · half a million · budget around 500K
timeline+20Q3 · next quarter · 下半年 · by year-end · for Q4
product+20具体产品/品类名称（ICU monitors、cardiac consumables）
strongIntent+20want to buy · wants demo · requested quote · evaluating vendors · 让我们出方案
weakIntent+10interested · curious · looking at · 感兴趣

评分规则：按 type 去重后求和，cap 100。分级：≥70 高（绿）· 40–69 中（黄）· <40 静默（不进 additionalActions）。 阈值调节直接改 prompt 文本即可，权重表也只在 prompt 里，没有运行时常量需要维护。

LLM 失守怎么办：Rule 6 用 "HIGHEST PRIORITY / MANDATORY" 强约束，Section 12.B 加 cross-ref 让它走 Rule 6 流程。E2E 已验证强弱两端表现稳定。若线上回归出现 <40 仍然输出的情况，下一步在 function-executor.ts 加入 defensive drop（读 _confidence，<40 则丢弃 draftOpportunity），但目前不预先加，避免遮蔽 prompt 问题。

07 / 风险与悬挂问题

提前列清楚，实施时不让它们变成意外

R1LLM 时态误判 → completed 活动被标 planned +

风险：模糊句式 "Visiting Rachel today" 模型可能判 planned 也可能判 completed。

缓解：在 Rule 9 强调"无明确完成/未来线索 → unspecified"。E2E 加歧义句用例，断言落到 unspecified。

R2检测卡频繁弹出造成骚扰 +

风险：销售可能每次拜访都聊到"客户感兴趣"，weakIntent 单独不上 40 分但累计可能上。

缓解：weakIntent 单条权重 10，无 amount/timeline/product 配合时最高 10 分，不会触发。阈值常量化便于上线后回调。

R3"该 account 是否有 open opp" 查询时延 +

风险：检测卡需要等查询返回才知道 Attach 按钮是否启用。慢网络下用户看到卡先出、按钮 200ms 后才亮起。

缓解：B-2 实施时 Attach 默认 disabled+loading，查询返回后切实际状态。或在 Activity Confirm 时预查询，并发隐藏延迟。

R4未来要不要把 temporalMode 推广到 Update Opportunity 等其他意图 +

本期范围仅 Activity。如果用户开始用"我们赢了这单 / 输了这单"等表达，Update Opportunity 走 stage 映射会很自然，但需要新增 mode → stage 的映射表与 prompt 例子。留待 I-9 评估。
