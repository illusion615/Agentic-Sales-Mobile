01 / Today's Architecture

今天的智能体长什么样：6 层 · 1 次 LLM · 16 条规则

先把今天能跑起来的东西完整描述一遍——没有美化也没有过度自责。这是后续讨论问题与重构方案的事实基础。

1.1 组件视图：6 层链路

1
Copilot Panel copilot-panel.tsx · ~965 行UI 入口：消息渲染、输入采集、IME 处理、语音/附件入口
2
Copilot Context copilot-context.tsx · ~1.9K 行状态拥有者：消息生命周期、连接管理、pending intent 注入、表单状态推进
3
Copilot Agent copilot-agent.ts · stateless意图层：一次 LLM 调用输出 function + arguments + matchTarget + additionalIntents
4
Function Executor function-executor.ts · ~1.4K 行执行层：40+ 函数分派（query / draft / fuzzy match / update + Copilot Studio 旁路）
5
Form Card form-card.tsx · ~1.2K 行用户闸门：草稿编辑 + selector + 创建入口（mutation 在此触发）
6
Data Layer generated/models + React Query持久化：Activity / Account / Contact / Opportunity 等实体
关键不变量
  • 每条用户消息恰好触发一次主决策 LLM 调用
  • 写库由前端在用户确认后执行（agent 不自动写）
  • Copilot Studio 只在 LLM 显式选 queryCopilotStudio 或解析失败时被调用，是旁路而非主链

1.2 当前意图识别的内部结构

Copilot Agent 的入口是 intentSystemPrompt（约 6000 字符），结构如下：

Persona

角色设定

「你是销售 CRM 智能助手」——非常薄，几乎只给了名字。

Rules

16 条规则清单

10+ 条 CRUD 规则 + 1 条「产品知识」规则 + 时态识别 + 模糊匹配 + 边界条件。结构性偏置严重倾向 CRUD。

Examples

7+ 条示例

少而集中在 draft activity；recommend / suggest / compare 等推理类动词几乎无样本。

Output

JSON 单输出

{ function, arguments, matchTarget?, additionalIntents? }——一次决定全部，无置信度，无 alternatives。

1.3 时态感知（tense-awareness）的现状

今天的 prompt 内嵌了一段时态识别逻辑，用于区分「记录已发生的拜访」vs「计划未来拜访」vs「创建未来商机线索」。这套逻辑被写死在 system prompt 文本里，没有独立测试入口、没有 telemetry，触发条件只有动词时态（「了 / 已 / 完成」vs「明天 / 准备 / 计划」）。

技术债标记

时态识别在新设计中将上提到 Step 1（Frame）作为「时间维度」的一个 axis，而非藏在动词词表里。原 tense-aware-activity-2026-05-17.html 文档的特性需求在本文 §5.3 重新表达。

02 / Why Intent Misfires

为什么意图会错：一个 case 揭开三个结构性问题

发现 bug 的过程是这样的：销售在商机详情页里说「为这个商机推荐一款产品」，系统打开了「记录拜访」的表单。把链路拆开后发现，错误不是 LLM 不够聪明，而是 prompt 自身的结构在引导它做错事。

Case #001 · 商机详情页 · 推荐产品
「为这个商机推荐一款合适的产品」
当前页面Opportunity Detail（商机 ID = OPP-0042）
实际意图基于商机的客户画像 + 历史活动，调用产品知识库给出推荐
LLM 输出{ "function": "draftActivity", "arguments": { "title": "推荐产品", "opportunityId": "OPP-0042" } }
结果错：打开了创建活动表单

2.1 三个结构性问题

问题 ① · Prompt 规则数量天然偏置 CRUD

16 条规则里有 10+ 条都在描述「什么时候应该 draft / update / query 一条记录」，只有 1 条规则说「产品相关问题用 queryCopilotStudio」。而且这条仅有的产品知识规则还被「页面门禁」限定为只在产品中心 / 产品详情页生效。

结果是：在商机详情页里说「推荐产品」时，LLM 看到的是 10+ 条 CRUD 规则的引力 + 0 条适用的知识规则，必然走 CRUD 分支。这不是「LLM 不够好」，是规则配比错了。

问题 ② · 把页面位置当成意图的硬约束

今天的 prompt 写法是「只有在 Product Center / Product Detail 页才路由到产品知识」。这把 UI 当成了意图的硬约束。但销售在商机页、客户页、活动页、甚至首页问产品问题都是合法场景。

正确做法：页面位置是「上下文绑定的暗示」，不是「意图的否决权」。

问题 ③ · 动词被当成意图信号，但动词本质是噪音

「推荐 / 建议 / 比较 / 应该」这些动词在销售对话里一词多义。看下面三个例子，动词全是「推荐」，意图却完全不同：

句子销售对象认知任务执行路径
推荐一款产品给这个商机Product知识检索 + 上下文匹配queryCopilotStudio + opportunity ctx
推荐这个商机的联系人Contact结构化 READ + ANALYZEgetContacts + rank by role
建议这个商机的跟进策略Strategy历史数据综合 + 知识库data fetch + queryCopilotStudio + synthesis

结论：动词不携带足够信号，关键是「宾语指向的销售对象」+「需要的认知任务」。

2.2 为什么正则/关键词补丁治标不治本

Rejected

方案 A · 加正则白名单

  • 动词「推荐/建议/比较」 → 强制走 queryCopilotStudio
  • 会误伤「推荐联系人」「比较活动历史」等结构化场景
  • 规则数量随宾语组合爆炸，维护成本指数级增长
Approved

方案 B · 重构意图识别架构

  • 放弃规则清单，让 LLM 按销售常识做三轴拆解
  • 把单次调用切成 Frame → Plan → Execute
  • 思考过程流式输出给用户，给「我们在帮你想」的临场感
  • 不确定时走结构化澄清，永不文本输入

03 / Redesign Principles

重构原则：把 prompt 写成销售专家的思维框架，而不是规则清单

所有原则都从一个判断出发：意图识别的本质是「按销售常识把一句话拆成可执行计划」，而不是「在词表里找匹配」。

P1销售对象 + 认知任务 决定意图

动词是噪音；宾语指向的销售对象（Product / Account / Contact / Opportunity / Activity / Strategy）才是意图的主信号。

P2销售常识 > 关键词匹配

Prompt 给模型一份「销售资深顾问」的角色框架与三轴拆解方法，而不是 16 条 if-then 规则。让模型可以泛化到任何未见过的句子。

P3思考可见 > 进度条

把 Frame → Plan 的中间结果以流式短句输出（「在判断你想要的是…」「在准备调用…」），用户得到「在帮我想」的临场感而非「在等」。

P4结构化澄清 > 猜

当 Plan 阶段置信度低或存在多个合理解读时，给出 2-3 个 chip 让用户点选，chip 上写明「点了会发生什么」。永不要求用户重新文本输入。

P5Golden cases > 穷举测试

保留 10-20 条「绝不能错的」典型 case 作为 prompt 重写的红线，配合生产 telemetry（澄清取消率、表单速关率）做持续改进。

P6可独立 rollback 的分期

每个阶段都可单独上线、单独回滚；任何一步出问题不影响其他步骤继续提供价值。

3.1 我们明确放弃的几件事

  • 放弃「在 prompt 里加规则数量」作为质量改进手段——规则越多，结构性偏置越严重
  • 放弃正则/关键词路由作为意图识别主路径
  • 放弃「页面位置 = 意图判别器」的写法
  • 放弃「单次 LLM 调用一把梭」——改为可观测、可中断、可澄清的多段
  • 放弃「无 telemetry 拍脑袋改 prompt」——改 prompt 必须挂在 golden cases + 在线指标上

04 / 3-Stage Pipeline

三段式 Pipeline：Frame · Plan · Execute

每一段都是独立的 LLM 调用（或快速本地推理），都有明确的输入、输出、耗时预算、流式 UX 状态、失败兜底。每段失败都能降级到上一段的结果。

t = 0 ms 用户消息进入 Copilot Context 附带：当前页面、当前实体 ID、最近 N 条对话历史
t = 0 – 400 ms Step 1 · Frame LLM-mini · streaming 拆解 3 轴：销售对象 / 认知任务 / 上下文绑定 → 流式短句「在判断你想要的是…」
t = 400 – 1200 ms Step 2 · Plan LLM-main · structured JSON 基于 Frame 结果，生成 function plan（function name + arguments + confidence + alternatives）→ 流式短句「在准备调用…」
t = 1200 ms + Step 3 · Execute or Clarify deterministic confidence ≥ 0.75 → 直接 Execute；否则 → 渲染澄清 chips（带「点了会发生什么」预览）

05 / Sales Domain Encoding

销售领域知识编码：三轴拆解 + 销售对象目录 + 认知任务目录

这一节给出 prompt 设计的「知识结构」：模型不靠规则、不靠正则，而是被赋予一份销售常识的本体（ontology）。任何句子都按三轴拆解；轴的取值来自有限目录。这把推理过程从「自由生成」收敛到「有界推理」，准确率上去、可观测性也上去。

5.3 上下文绑定（Context Binding）+ 时间维度

原 tense-aware-activity-2026-05-17.html 文档里关于「时态识别」的特性需求，在新设计里被上提为 Frame 阶段的两个独立信号：

A
Context Binding当前页面 / 当前实体 ID / 最近 N 轮对话引用 → 决定「这条消息提到的它，指的是哪一个」
B
Time Axispast（记录已发生）/ now（当前态）/ future（计划）→ 影响 Activity / Opportunity 的写入语义
文档合并说明

原 tense 文档要求「记录已发生的活动需要把 startTime 设为过去时间，并跳过提醒；计划未来的活动需要设为未来时间并启用提醒」。这条契约在本文 §6（Plan 阶段输出契约）保留，并由 §8 golden cases 中两条 case 守护。

5.4 Step 1 (Frame) System Prompt 草稿

# Frame Prompt（节选，~1200 字符目标，远小于今天的 6000）

You are a senior B2B sales consultant helping a field rep decide
what they actually mean when they speak into the CRM app.

Your job: decompose the user's utterance along three axes:

1. SALES OBJECT — which of: Account, Contact, Opportunity,
   Activity, Product, Strategy, External. (The noun phrase the
   user is asking about — NOT the verb they used.)
2. COGNITIVE TASK — which of: READ, WRITE, ANALYZE,
   RECOMMEND, EXPLAIN, SYNTHESIZE.
3. CONTEXT BINDING — does “this/it/that” refer to the
   current page's entity? Are recent turns relevant?
   Also: time axis (past / now / future).

Heuristics from senior reps:
- “Recommend a product” → SALES OBJECT = Product, TASK = RECOMMEND,
  no matter which page you're on.
- “Recommend a contact” → SALES OBJECT = Contact, TASK = ANALYZE
  (rank existing contacts by role/recency).
- “Suggest a strategy” → SALES OBJECT = Strategy, TASK = SYNTHESIZE.
- A bare verb (“log”, “add”, “update”) with no object → ask context
  binding; if still ambiguous, return low confidence.

Return strict JSON:
{ "object": ..., "task": ..., "context": {...}, "time": "past|now|future", "confidence": 0.0–1.0 }

注意这份 prompt 没有「if-then 规则清单」，也没有「函数名」——Frame 只关心拆解，不关心调哪个函数。函数选择交给 Step 2。

06 / Visible Thinking UX

流式思考 UX：让用户「看到」系统在帮他想

多段调用会引入额外延迟（Frame + Plan ≈ 1.2 秒，比单次 0.8 秒慢 50%）。补偿方式不是优化延迟本身，而是把等待时间变成「正在思考」的可见过程——感知时长比真实时长更短。

6.2 UI 示意

为这个商机推荐一款合适的产品
在判断你想要的是「为这条商机找产品推荐」… 在准备调用产品知识库，带上这条商机的客户画像… 推荐 3 款（按匹配度排序）：1) ProductA — 适配客户行业；2) ProductB — 历史购买偏好；3) ProductC — 同段位案例…

6.3 文案设计原则

  • 第一人称 + 现在进行时（「在判断…」「在准备…」），让用户觉得有「同事在帮他想」
  • 把识别出的「销售对象 + 任务」用自然语言复述一次，让用户能立即发现误解（早期纠错）
  • 禁止「正在加载…」「Loading…」这种空文案——用户看了等于没看
  • 每条 thought 行 ≤ 30 字（zh）/ 60 chars（en），保证单手单屏可读

07 / Structured Clarification

澄清交互：当 Plan 不确定，用 chip 而不是问句

今天的兜底是「让用户重说一遍」——已经被证明是体验最差的兜底，因为用户不知道改哪里。新设计在 Plan 阶段一次性产出 alternatives，置信度不足时把 2-3 个选项渲染成 chip，每个 chip 标明「点了会发生什么」。

7.1 chip 设计契约

字段类型说明
labelstring ≤ 16 字主标题，描述「你想要的」
previewstring ≤ 30 字副标题，说明「点了会发生什么」
planPlan output预生成好的 function 调用，点击后直接进入 Execute
iconenum可选；用于强化「这是哪类操作」

7.2 实例

跟我说一下这个客户
我看到两种可能，你想要哪种？ ① 看客户档案跳到客户详情页，展示基础信息 + 联系人 ② 看最近活动按时间线列出最近 5 次拜访/电话/邮件 ③ 综合分析结合画像 + 活动 + 商机给出现状总结

7.3 规则

  • chip 数量恒定 ≤ 3 条；多了会让用户更迷茫
  • chip 之间必须意图正交（不要给「看客户详情」和「看客户基础信息」这种近义选项）
  • 永远提供「都不是 → 重新说」的兜底入口（小字 link 而非 chip）
  • chip 取消 / 跳过都要打 telemetry（澄清取消率是 prompt 质量的核心指标）

08 / Quality Loop

质量闭环：Golden Cases + 在线 Telemetry

LLM 系统不可能靠穷举测试集保证质量。我们用「10-20 条不能错的红线 case」+「3 个可观测的在线指标」来兜住质量，配合每次 prompt 改动的回归扫描。

8.1 Golden Cases（初版 12 条）

#用户句子页面 / 上下文正确输出（Frame）
1为这个商机推荐一款产品Opportunity DetailProduct / RECOMMEND

3建议这个商机的跟进策略Opportunity DetailStrategy / SYNTHESIZE
4比较这两个联系人的跟进历史Account DetailActivity / ANALYZE

任何 prompt 改动必须先在这 12 条上验证；任一条错就阻塞合并。Case 集合随 production telemetry 持续增长（每发现一种新错法就加一条）。

8.2 在线 Telemetry（3 个核心指标）

Metric 1clarify-cancel-rate

澄清 chip 被点「都不是」或取消的比例。> 15% 说明 Plan 阶段 alternatives 不准。

Metric 2form-quick-close-rate

表单打开后 < 3 秒被关闭（用户判断打开的是错的表单）。> 8% 说明 Execute 选错函数。

Metric 3retry-within-30s-rate

同一用户 30 秒内对同一上下文重发消息（含改写）。> 10% 说明响应「不对路」。

埋点位置

所有指标在 Copilot Context 里统一打点，输出到 AgentLog 表（已在 .agent/plans/sales-copilot-mobile.md 中预留）。看板暂用 Dataverse + Power BI；后续可上 App Insights。

09 / Roadmap

实施路线图：4 个独立可上线阶段

每个阶段都能单独发上去给用户用、单独 rollback、单独度量。优先级按「最大价值 / 最小风险」排。

P1
Phase 1 · Frame Prompt 独立化（最小可上线单元）抽出 Frame prompt，旧 prompt 保留为 Plan 阶段；让两段并存，Frame 结果只用于打 telemetry（影子模式）。验证拆解准确率 ≥ 90% 后切换。
P2
Phase 2 · Plan Prompt 重写 + confidence 输出Plan 收 Frame 的输出，输出 { function, arguments, confidence, alternatives }。confidence 不达阈值时仍按今天的方式继续（不触发 Clarify UI），先验证置信度分布合理。
P3
Phase 3 · 流式思考 UX + Clarify chips开启流式短句状态机；开启 confidence < 阈值时的 Clarify chips。这一阶段是用户最直接感知的变化。
P4
Phase 4 · Telemetry 看板 + Golden-case 自动回归三个核心指标的 Power BI 看板；CI 上接 golden-case 回归脚本（每次 prompt 改动跑 12 条）。形成质量闭环后可加速 prompt 迭代。

10.1 影子模式（Shadow Mode）契约

P1 阶段两条管道并存：旧 prompt 主路 + 新 Frame prompt 影子。影子结果不影响 UI，只写入 AgentLog。这让我们用真实流量校准 Frame 的拆解准确率，而不是靠合成数据。

切换判据
  • 影子模式累计 ≥ 500 条真实样本
  • Frame 与旧 prompt 在 SALES OBJECT 上一致率 ≥ 90%
  • 所有 12 条 golden cases 在 Frame 上通过
  • 新增 cost 月度 ≤ 老板预设阈值

11 / Open Questions

开放问题：本文件不强答的事

工程上不能回避的判断，但今天信息不够，下次评审解。每个问题给出当前倾向 + 决策时机。

Q1 · Frame 用 mini 模型还是主模型？

倾向：mini 模型（成本 1/10，延迟 1/3，准确率预期 85%+ 即可）。
决策时机：P1 影子模式跑满 500 条后看真实数据。

Q2 · confidence 阈值定多少？

倾向：0.75。
依据：低于 0.7 时 alternatives 显著不正交；高于 0.8 时澄清场景几乎触发不了。
决策时机：P2 上线 1 周后看 confidence 分布直方图。

Q3 · 流式协议用 SSE 还是 WebSocket？

倾向：SSE。
决策时机：P3 开 UX 改造时确定。

Q4 · 何时把主模型从 GPT-4 切到 Foundry/自托管？

倾向：不在本架构范围内决策。当月活到一定规模、成本曲线明显时再单独立项评估。本文 Plan 阶段对模型解耦（输入输出契约稳定），切换不需要改架构。
